# Skill：AI日历 Vue3 全局开发规范

## 基础信息

- **Skill 名称**：AI Calendar Vue3 开发规范
- **描述(Description)**：适用于可爱风AI智能日历项目，统一约束代码编写、组件开发、样式设计、文件管理等全流程规则。要求新增功能时不得修改已有完成代码，保证项目编码风格、视觉风格、目录结构全局统一，提升代码可维护性与项目一致性。
- **使用场景**：编写日历新功能、开发组件、封装工具方法、样式调整、逻辑优化、问题修复等所有代码生成与迭代场景。
- **使用时机**：本项目所有编码相关需求，执行代码输出前必须启用并严格遵守本规范。
- **技术栈**：Vue 3 + Vite + Pinia + SCSS
- **视觉风格**：全局可爱风

## 一、核心强制约束

1. 开发新功能、新增代码时，**禁止修改、删除、注释已完成的代码、组件、样式及业务逻辑**，仅独立新增代码/文件。
2. 所有代码遵循 Vue3 官方标准规范与前端通用工程化规范，代码简洁易读，复杂逻辑添加必要注释。
3. 优先复用现有组件、工具函数、状态方法，避免重复开发。

## 二、目录与文件规范

1. 目录划分：页面文件存放于 `views`，通用组件存放于 `components`，工具函数存放于 `utils`，全局样式存放于 `styles`。
2. Vue 组件文件：采用**大驼峰(PascalCase)** 命名，例：CalendarHeader.vue、DateItem.vue。
3. 工具/配置文件：采用小驼峰命名。
4. 样式统一使用 SCSS，组件样式必须添加 `scoped` 私有化。

## 三、代码语法与命名规范

1. 所有 Vue 组件仅使用 `<script setup>` 语法，禁用 Options API。
2. 变量、函数、方法：**小驼峰(camelCase)** 命名，语义化命名，见名知意。
3. 常量：全大写 + 下划线分隔，例：DEFAULT_YEAR、MAX_SCHEDULE_COUNT。
4. Pinia 状态仓库：统一使用 `useXxxStore` 格式命名。
5. CSS/SCSS 类名：**短横线(kebab-case)** 命名，例：calendar-container、schedule-card。
6. 页面事件处理函数：统一以 `handle` 为前缀，例：handleSelectDate、handleAddSchedule。

## 四、响应式布局与单位规范（强制）

1. 尽量**少使用固定 px 单位**，优先使用响应式单位：rem / em / rpx / % / vw / vh。
2. 项目统一使用一套响应式单位标准，确保多端、多尺寸屏幕自适应。
3. 布局优先使用 Flex、Grid 等弹性布局，禁止写死固定宽高导致无法适配。
4. 字体、间距、圆角、边框等，优先使用相对单位，保证全局缩放一致性。
5. 禁止大量使用固定 px 硬编码，避免在不同设备上出现错位、溢出、显示异常。

## 五、可爱风样式统一规范

1. 圆角：所有容器、卡片、按钮统一使用 8px ~ 16px 圆角，禁止直角设计。
2. 阴影：使用低透明度轻柔阴影，禁用厚重深色阴影。
3. 配色：采用低饱和度马卡龙柔和色系，全局配色保持统一。
4. 图标与装饰：选用圆润卡通风格元素，贴合可爱视觉定位。

## 六、业务与交互规范

1. 新增功能交互逻辑，需与项目原有交互逻辑、操作习惯保持一致。
2. 数据请求、数据处理逻辑统一封装至 `utils` 或 Pinia Store，组件仅负责视图渲染。
